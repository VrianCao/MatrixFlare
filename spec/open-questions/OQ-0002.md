## Question

Cloudflare 当前关于 `TEST-COST-001` / `EVID-COST-001` 所依赖的成本证据，有两类尚未收敛的官方语义：

1. Workers tracing / OpenTelemetry 文档在 `2026-03-26` 观察时存在互相冲突的计费口径：

* `workers-opentelemetry` 页面写明：自 `2026-03-01` 起，Workers Paid 的 traces 与 logs 各含 `10M` events / month，超额 `$0.05 / million additional events`。
* `workers-traces` 页面写明：tracing spans 作为 observability events，与 Workers Logs 共享 monthly quota / pricing / retention，并展示 Paid `10M included / month + $0.60 per additional million events, retention 7 days` 的表述。
* `workers-pricing` / `workers-logs` 页面则把 Workers Logs 写成 Paid `20M log events / month + $0.60 / additional million, retention 7 days`。

因此，当前无法从官方文档中唯一确定以下问题：

* traces 是否与 exported logs 共用 included quota；
* traces 的 overage price 到底是 `$0.05 / million` 还是 `$0.60 / million`；
* traces retention 是否与 Workers Logs 的 `7 days` 完全等同，还是遵循另一套 observability 产品语义。

2. `2026-04-05` 对官方 Cloudflare billing docs 的重新核对，又把 production-side cost contract 收敛到比 earlier TODO 更精确的现实：

* 官方 usage-based billing 文档明确说明：Cloudflare usage charges 依赖账户 billing date，对应 invoice 覆盖的是 previous billing period，而不是保证与 calendar month 对齐。官方示例甚至明确给出“billing date = March 15 时，April 15 invoice 覆盖 March 16 到 April 15”的窗口。这意味着仓库先前把 “上一个自然月” 当成默认 truthful production snapshot window 的做法站不住。
* 官方 account billing profile API 文档仍暴露 `next_bill_date`，因此当前自动化至少可以、也应该优先用它解析 latest closed billing period，而不是继续用 shell `date` 脑补 previous calendar month。
* Cloudflare 官方 account subscriptions API 另外暴露 `current_period_end`，其语义同样是当前周期结束 / next billing due；这意味着 billing profile `next_bill_date` 不是唯一可能的官方 billing-cycle anchor。
* 但这并没有关闭 production blocker，因为官方 Billing Usage / PayGo API 仍是 beta/select-account access，而且 current target account/workflow 尚未证明它可用来产出真实 monthly usage rows。

3. `2026-04-05` 的最新 same-head remote truth 又把 production-side blocker 继续改写成了比 earlier rerun 更精确的现实：

* latest reviewed pushed head `700b8600264b78eacc07733a7fd09248d9c68e8a` 继承了 earlier same-head `prod-install` run `24000789563` 已建立的 fixed-name production topology：`matrix-gateway-worker-prod`、`matrix-jobs-worker-prod`、`matrix-ops-worker-prod`、workers subdomain `yevdndjyebksug`，以及 Access-protected prod ops URL `https://matrix-ops-worker-prod.yevdndjyebksug.workers.dev`。这说明“规范固定命名的 production topology 仍不存在”已不再是 active blocker。
* same-head standalone `prod-cost-monthly` run `24003512022` 与 embedded `prod-cost` job in run `24003512030` 现在都先执行到了 revised entry contract 中的两个关键前置路径：根据官方 billing profile API 尝试解析 billing window、以及下载当前 prod baseline `ProdInstallRecord`。其中 billing-window step 在缺失可信 `next_bill_date` 时按 fail-closed 语义写出 blocker；standalone workflow 继续保留并上传 raw blocker bundle；embedded `prod-cost` job 则在同轮 `nonlocal-phase08` 中进一步证明 workflow 也能在 fail-closed gate 前进入 `Write prod-cost provenance` step。
* 但当前 active blocker 已不再是 earlier run `24000831481` / `24001049145` 那种“先打到 `/billing/usage/paygo` 再收到 `404 page not found`”。在 latest reviewed pushed head `700b860` 的 same-head rerun 中，官方 billing profile request 本身成功返回了 profile payload，但 `result` 里没有 `next_bill_date`；于是 `billing-window.json` 被诚实写成 `resolution_method = cloudflare-account-billing-profile-next-bill-date` + `error = "Cloudflare billing profile must include a valid next_bill_date"`，workflow 在进入 usage query 之前就 fail-closed 停止。
* `2026-04-05` current same-head remote rerun `24005804747`（head `a706c04127a5cd7978b850061c81b3c646220a4e`）又把这条 contract 再向前推进了一步：在同一 target account 上，`Resolve closed production billing window` 已真实从 billing profile 缺字段的情况退回到唯一的 account subscriptions `current_period_end`，`billing-window.json` 现已写出 `resolution_method = cloudflare-account-subscriptions-current-period-end`、`next_bill_date = 2026-04-29T00:00:00Z`、`from_date = 2026-03-01`、`to_date = 2026-03-29`；这证明 subscriptions fallback 已拿到 same-head remote proof，而不再只是本地 parser/tooling closure。
* 同一 run `24005804747` 中，`Capture production cost snapshot` 继续按 fail-closed 语义停止，并在 raw artifact 保留后明确报出：`production cost snapshot requires a closed billing window that starts after prod install date 2026-04-05`。这说明 production-side active blocker 已从“缺可信 billing-cycle anchor”进一步收敛成“当前账号最新 closed billing period 仍早于 fixed prod install date，因此还没有 truthful post-install closed billing period”。

因此，当前也无法从官方资料与同轮 remote truth 中唯一确定以下问题：

* 哪个官方 billing surface 可以在不降标的前提下稳定产出 fixed production topology 的 monthly cost snapshot + model comparison；
* 在 account billing period 不等于 calendar month 的前提下，仓库应如何把官方 billing-cycle anchor（billing profile `next_bill_date` 或 subscriptions `current_period_end`）精确归一化成 latest closed billing period query window，并在官方 surface 返回 records 时继续做 truthful validation；
* 项目是否必须等待 `2026-04-05` prod install 之后的第一个完整 post-install closed billing period，才能诚实生成 `ProdCostSnapshotAttestation`；
* 如果官方 billing automation 继续不可用，`EVID-COST-001` 是否只能继续保持 fail-closed，或改由新的 `DEC-ID` 明确接受受控人工导出/审查，而不能被 GitHub Actions 全自动闭环。

## Why It Matters

* [13-cloudflare-platform-constraint-register.md](/root/Matrix/spec/framework/13-cloudflare-platform-constraint-register.md) 需要为 `CF-WKR-017` 提供可审计、可追责的平台事实；若这里写错，后续 [41-observability-performance-and-cost.md](/root/Matrix/spec/framework/41-observability-performance-and-cost.md) 的 Included Quotas Matrix 与成本门禁都会被污染。
* [41-observability-performance-and-cost.md](/root/Matrix/spec/framework/41-observability-performance-and-cost.md) 的 `REQ-OPS-003`、`TEST-COST-001` 与 `EVID-COST-001` 依赖稳定的计费单元定义；当前若把某一份页面当作确定真相，会让预算和发布门禁出现系统性偏差。
* Phase 08 现在已经把 non-local harness / attestation / GitHub Actions 接线推进到“可以生成外部 provenance”的阶段，但 `EVID-COST-001` 仍要求真实 production monthly cost snapshot；即使 same-head `prod-install` 已建立 fixed-name prod topology、workflow 也已能保留并上传 raw blocker bundle，只要官方 billing surface 仍未对当前账号/workflow 证明可用，且 post-install closed month 还不存在，就不能把“workflow 能跑起来并上传 bundle”误报成成本证据已自动化。
* `persist = true` / `false` 的选择也会受到 retention / dashboard 存储语义影响；若 quota / pricing 口径不清，团队很容易误判“启用 traces 是否只增加一个小的外发成本”。

## Options

1. Use the OTel export page as the sole pricing truth
   把 traces/logs 各 `10M` included + `$0.05 / million` 视为当前准则，忽略 Traces 页面。

2. Use the Traces page as the sole pricing truth
   把 trace spans 视为与 Workers Logs 共享 observability event quota / pricing / retention，忽略 OTel export 页面。

3. Keep pricing semantics unresolved in the normative baseline
   仅保留双方都不冲突的最小事实，把 trace quota/pricing/retention 视为 open question，在规范与成本门禁中参数化处理，直到 Cloudflare 文档或官方答复收敛。

4. Keep production monthly cost snapshot automation unresolved until target-account billing access and a truthful post-install closed month both exist
   在 `EVID-COST-001` 中继续要求真实 production monthly snapshot，但不宣称 GitHub Actions 已能自动抓取它；只有当目标账号的官方 billing surface 被验证可用，且 fixed prod topology 已运行到至少一个 truthful post-install closed billing period，或有受控人工导出流程进入 evidence contract，才允许关闭该自动化问题。

## Current Recommendation

当前建议采用 Option 3 + 4：

* 在规范基线中保留 `CF-WKR-018` 的 `persist` 默认值事实；
* 在 `CF-WKR-017` 中显式登记“Cloudflare observability docs conflict”；
* 在成本模型中把 `trace_span_count`、`exported_log_event_count` 与 `persist_enabled` 分开计量；
* 在 `TEST-COST-001` / `EVID-COST-001` 中避免把单一 tracing included quota / overage price 写死；
* 在 target production account 尚未同时满足“官方 billing snapshot surface 已被当前 workflow 证明可用”与“fixed prod topology 已运行到至少一个 truthful post-install closed billing period”之前，不把 `EVID-COST-001` 标成“可由 GitHub Actions 全自动闭环”；
* 把 runs `24000789563`,`24003512022`,`24003512030`,`24005242689`,`24005804747` 视为当前 active truth：fixed-name prod topology 已存在，revised prod-cost entry contract 也已 same-head 验证 raw blocker retention、billing-window resolution path 与 prod baseline download；其中 `24005242689` 证明当前账号在新增 billing-view scope 后仍可能缺失 profile `next_bill_date`，`24005804747` 则进一步证明 workflow/tooling 已能在 same-head remote truth 下退回到官方 subscriptions `current_period_end` 并解析 closed billing window。当前 production-side 剩余 blocker 也因此收敛为单点事实：最新 truthful closed billing period 仍覆盖 `2026-04-05` prod install 之前的时间，所以还不能生成 truthful `ProdCostSnapshotAttestation`；
* 若某次发布必须依赖 tracing 成本上限，则在 release gate 前重新核对官方文档，必要时补充人工 support confirmation 或新的 `DEC-ID`。

## Closure Criteria

本问题关闭前，必须至少满足以下条件之一：

* Cloudflare 官方文档收敛为一致口径，且 `CF-WKR-017`、`41`、相关成本测试已同步更新；
* Cloudflare 官方支持或合同文档给出针对目标账户/usage model 的明确书面口径，并被纳入可审计 evidence；
* 目标生产账号的官方 billing snapshot API / export surface 已经被证明可用于产出真实 monthly cost snapshot，且规范固定命名下的 production topology 已真实存在并运行到至少一个 post-install closed billing period，可以生成 `ProdCostSnapshotAttestation`；或项目通过新的 `DEC-ID` 接受受控人工导出流程并把其审计链写入 evidence contract；
* 项目通过新的 `DEC-ID` 明确采用某一官方口径，并在成本门禁中承认其风险与适用边界。
