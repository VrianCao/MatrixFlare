## Question

Cloudflare 当前关于 `TEST-COST-001` / `EVID-COST-001` 所依赖的成本证据，有两类尚未收敛的官方语义：

1. Workers tracing / OpenTelemetry 文档在 `2026-03-26` 观察时存在互相冲突的计费口径：

* `workers-opentelemetry` 页面写明：自 `2026-03-01` 起，Workers Paid 的 traces 与 logs 各含 `10M` events / month，超额 `$0.05 / million additional events`。
* `workers-traces` 页面写明：tracing spans 作为 observability events，与 Workers Logs 共享 monthly quota / pricing / retention，并展示 Paid `10M included / month + $0.60 per additional million events, retention 7 days` 的表述。
* `workers-pricing` / `workers-logs` 页面则把 Workers Logs 写成 Paid `20M log events / month + $0.60 / additional million, retention 7 days`。

因此，当前无法从官方文档中唯一确定以下问题：

* traces 是否与 exported logs 共用 included quota；
* traces 的 overage price 到底是 `$0.05 / million` 还是 `$0.60 / million`；
* traces retention 是否与 Workers Logs 的 `7 days` 完全等同，还是遵循另一套 observability 产品语义。

2. `2026-04-01` 观察时，Cloudflare Billing Usage/Paygo API 虽已有官方文档，但仍带有 beta / select-account 适用边界；与此同时，latest reviewed pushed-head run `23993238210`（head `d7f8ab91d3feaad29d161ba27086e12c4c42b665`）已把这条问题的真实形状进一步收窄：

* 目标账号现已证明具备足以让 GitHub Actions 调用 `GET /billing/usage/paygo?from=2026-03-01&to=2026-03-31` 的 API 访问能力；同轮 raw `prod-cost` job 也已真实完成 immutable R2 bundle upload 到 `r2://matrix-evidence-prod/gha/23993238210/1/prod/20260405T031704Z/cost-bundle.tgz`。这说明此前围绕“Billing API scope 可能不足”与“`matrix-evidence-prod` bucket 缺失”的 blocker 已不再是 active root cause。
* 但同一 run 也同时证明，当前 target account 仍不存在规范固定命名下的 production Worker topology：`Capture production cost snapshot` 直接在 `GET /workers/scripts/matrix-gateway-worker-prod/deployments` 上收到 `404 / code 10007 / "This Worker does not exist on your account."`，随后 `prod-deployment-identity.json` 无法生成，`prod-cost-provenance` 也因缺 deployment identity fail-closed。

因此，当前也无法从官方资料与同轮 remote truth 中唯一确定以下问题：

* 在 API access 已被证明可用的前提下，项目是否必须先建立并持续运行符合 Spec 命名/拓扑的真实 production environment，直到至少一个完整 calendar month 闭合，才能诚实生成 `ProdCostSnapshotAttestation`；
* 如果短期内仍不存在该 production topology，`EVID-COST-001` 是否只能继续保持 fail-closed，或改由新的 `DEC-ID` 明确接受受控人工导出/审查，而不能被 GitHub Actions 全自动闭环；
* 哪个官方 billing surface 可以在不降标的前提下稳定产出 production monthly cost snapshot + model comparison。

## Why It Matters

* [13-cloudflare-platform-constraint-register.md](/root/Matrix/spec/framework/13-cloudflare-platform-constraint-register.md) 需要为 `CF-WKR-017` 提供可审计、可追责的平台事实；若这里写错，后续 [41-observability-performance-and-cost.md](/root/Matrix/spec/framework/41-observability-performance-and-cost.md) 的 Included Quotas Matrix 与成本门禁都会被污染。
* [41-observability-performance-and-cost.md](/root/Matrix/spec/framework/41-observability-performance-and-cost.md) 的 `REQ-OPS-003`、`TEST-COST-001` 与 `EVID-COST-001` 依赖稳定的计费单元定义；当前若把某一份页面当作确定真相，会让预算和发布门禁出现系统性偏差。
* Phase 08 现在已经把 non-local harness / attestation / GitHub Actions 接线推进到“可以生成外部 provenance”的阶段，但 `EVID-COST-001` 仍要求真实 production monthly cost snapshot；即使 billing API access 已在 run `23993238210` 中被证明可用，只要目标账号里还没有规范固定命名下的 production Worker topology，就不能把“workflow 能调 API 并上传 bundle”误报成成本证据已自动化。
* `persist = true` / `false` 的选择也会受到 retention / dashboard 存储语义影响；若 quota / pricing 口径不清，团队很容易误判“启用 traces 是否只增加一个小的外发成本”。

## Options

1. Use the OTel export page as the sole pricing truth
   把 traces/logs 各 `10M` included + `$0.05 / million` 视为当前准则，忽略 Traces 页面。

2. Use the Traces page as the sole pricing truth
   把 trace spans 视为与 Workers Logs 共享 observability event quota / pricing / retention，忽略 OTel export 页面。

3. Keep pricing semantics unresolved in the normative baseline
   仅保留双方都不冲突的最小事实，把 trace quota/pricing/retention 视为 open question，在规范与成本门禁中参数化处理，直到 Cloudflare 文档或官方答复收敛。

4. Keep production monthly cost snapshot automation unresolved until target-account capability is proven
   在 `EVID-COST-001` 中继续要求真实 production monthly snapshot，但不宣称 GitHub Actions 已能自动抓取它；只有当目标账号的官方 billing surface 被验证可用，或有受控人工导出流程进入 evidence contract，才允许关闭该自动化问题。

## Current Recommendation

当前建议采用 Option 3 + 4：

* 在规范基线中保留 `CF-WKR-018` 的 `persist` 默认值事实；
* 在 `CF-WKR-017` 中显式登记“Cloudflare observability docs conflict”；
* 在成本模型中把 `trace_span_count`、`exported_log_event_count` 与 `persist_enabled` 分开计量；
* 在 `TEST-COST-001` / `EVID-COST-001` 中避免把单一 tracing included quota / overage price 写死；
* 在 target production account 尚未同时满足“官方 billing snapshot API access 已证明可用”与“规范固定命名下的 production topology 已真实存在并可生成 deployment identity”之前，不把 `EVID-COST-001` 标成“可由 GitHub Actions 全自动闭环”；
* 把 run `23993238210` 视为当前 active truth：Billing API scope 与 prod evidence bucket 已不再是 blocker，但 `matrix-*-prod` Worker topology 缺失仍会让 `ProdCostSnapshotAttestation` fail-closed；
* 若某次发布必须依赖 tracing 成本上限，则在 release gate 前重新核对官方文档，必要时补充人工 support confirmation 或新的 `DEC-ID`。

## Closure Criteria

本问题关闭前，必须至少满足以下条件之一：

* Cloudflare 官方文档收敛为一致口径，且 `CF-WKR-017`、`41`、相关成本测试已同步更新；
* Cloudflare 官方支持或合同文档给出针对目标账户/usage model 的明确书面口径，并被纳入可审计 evidence；
* 目标生产账号的官方 billing snapshot API / export surface 已经被证明可用于产出真实 monthly cost snapshot，且规范固定命名下的 production topology 已真实存在并运行足够长的闭合月度窗口，可以生成 `ProdCostSnapshotAttestation`；或项目通过新的 `DEC-ID` 接受受控人工导出流程并把其审计链写入 evidence contract；
* 项目通过新的 `DEC-ID` 明确采用某一官方口径，并在成本门禁中承认其风险与适用边界。
