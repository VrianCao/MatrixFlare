## Question

`TEST-OPS-001` 要求 pre-release 非本地门禁真实证明 `new Worker -> old DO` 与 `old Worker -> new DO` 兼容性。到 `2026-04-04` 为止，dual-version deployment contract、dedicated canonical suite、strict evidence mapping 和 Access-protected `/_ops` ingress 都已经落地，本问题也不再是“contract 还没写完”。

latest pushed-head run `23981099390`（head `2e3bde8ee183744192e7a27ae5fc4b77f5f2af89`）进一步把 blocker 收窄到了一个更具体的 current-head runtime observation gap，而不再只是 earlier `1042` same-zone transport hypothesis：

1. [cloudflare-workers-versions-deployments.md](/root/Matrix/research/sources/cloudflare-workers-versions-deployments.md) 与 [cloudflare-workers-gradual-deployments.md](/root/Matrix/research/sources/cloudflare-workers-gradual-deployments.md) 已经固定了当前 rollout contract：
   * Worker versions 与 deployments 分离；
   * gradual deployment 中 Durable Object version assignment 由 deployment percentages 决定；
   * `Cloudflare-Workers-Version-Key` / `Cloudflare-Workers-Version-Overrides` 只控制请求命中哪个 Worker version，不会改写已分配给某个 DO 的 version assignment。
2. 同一轮官方 docs 还给出两条仍然成立的平台事实：
   * [cloudflare-workers-limits.md](/root/Matrix/research/sources/cloudflare-workers-limits.md) 明确说明：使用全局 `fetch()` 在同一 zone 内调用另一个 Worker、且未使用 service binding 时会失败；
   * [cloudflare-workers-compatibility-flags.md](/root/Matrix/research/sources/cloudflare-workers-compatibility-flags.md) 明确说明：只有启用 `global_fetch_strictly_public` 后，同 zone public request 才会按“来自公共 Internet”的方式重新进入 Cloudflare front door；否则会被路由到 zone origin，并绕过映射到该 URL 的 Worker 与 Cloudflare security settings。
3. earlier run `23980399903` 暴露的 `404 text/plain "error code: 1042"` 仍然重要，因为它解释了为什么 repo 必须先把 `ops-worker` 钉到 `global_fetch_strictly_public`；但 latest pushed-head run `23981099390` 已不再在这一步失败，这意味着 current transport repair 至少足以回到真实 gateway Matrix JSON。
4. run `23981099390` 的 raw pre-release artifact 给出的 current-head failure shape 现在是：
   * `tests/pre-release/test-ops-001.test.mjs` 与 `tests/pre-release/test-cost-001.test.mjs` 都确实被执行；
   * `TEST-COST-001` 继续按 [OQ-0006](/root/Matrix/spec/open-questions/OQ-0006.md) 真实 `500/internal` fail-closed；
   * `TEST-OPS-001` 则在 rollout probe 的第一跳 `POST /_matrix/client/v3/register` 上返回 `409 precondition_failed`，消息为 `Gateway version override mismatch ... requested <Cloudflare version UUID>, observed gha-pre-release-...-gateway-worker`；
   * `rollout_skew_probe` sidecar 因而缺失，pre-release attestation 被 fail-closed 跳过，`EVID-OPS-001` 继续失败。
5. 结合 repo 当前实现再复核后，remaining gap 已不再是 “transport 还没修”：
   * `apps/gateway-worker/src/index.mjs` 当前已经会同时暴露 `x-matrix-rollout-probe-gateway-version-id` 与 `x-matrix-rollout-probe-gateway-version-tag`；
   * 但 `packages/control-plane/src/ops-handler.mjs` 仍只把 request 中的 Cloudflare version UUID 与 `observed_gateway_version_id` 做 strict equality，完全没有消费 official version tag；
   * latest pushed-head 的 `observed gha-pre-release-...-gateway-worker` 也说明：repo 不能再把本地 `WORKER_VERSION_ID` fallback 冒充成 attested Cloudflare version ID。

因此，较早把 latest blocker 继续写成 “same-zone transport 未闭合” 已经不够准确。当前更贴近事实、且受官方 docs 强支撑的结论是：

* `IF-OPS-009` / `DATA-OPS-012` / `FLOW-OPS-ROLLOUT-SKEW` 已经达到 `SR3` 所需的大部分资产；
* current active blocker 已收敛为 runtime observation contract：request 仍应使用 Cloudflare version UUID 做 explicit targeting，但 response 侧必须以 official version ID 或 official version tag 完成 fail-closed override 观测；
* repo-local `WORKER_VERSION_ID` fallback 只可作为本地诊断，不得再被当作 attested Cloudflare version identity。

在新的 pushed-head GitHub Actions run 真实完成“修正后的 runtime identity contract（official version ID + official version tag）+ dual-version probe + attestation generation + `evidence-l1` 消费”之前，`TEST-OPS-001` / `EVID-OPS-001` 仍不得转为 `pass`。

## Why It Matters

* [42-deployment-migration-and-recovery.md](/root/Matrix/spec/framework/42-deployment-migration-and-recovery.md) 已把 `REQ-OPS-010`,`REQ-OPS-011`,`REQ-OPS-012` 固定为 rollout compatibility gate，而不是 deployment smoke。
* [43-testing-and-compliance.md](/root/Matrix/spec/framework/43-testing-and-compliance.md) 把 `TEST-OPS-001` 定义为 pre-release 非本地门禁；如果没有真实 dual-version orchestration，就不能再把 “healthz 正常” 或 “active deployment identity 已回读” 误写成 version skew proof。
* [44-verification-and-evidence-register.md](/root/Matrix/spec/framework/44-verification-and-evidence-register.md) 要求 `EVID-OPS-001` 是 rollout compatibility attested pre-release report；当前若不把 contract gap 显式化，后续很容易继续把 placeholder suite 写成“已有门禁”。

## Options

1. Repair the rollout contract and implementation, then rerun attested pre-release proof
   在保留现有 dual-version deployment + bounded sampling contract 的前提下，把 current-head 的 runtime observation contract 修正到符合 Cloudflare 官方 docs 的路径：request 继续使用 Cloudflare version UUID targeting，response 改为 official version ID 或 official version tag 校验，并禁止 repo-local fallback 伪装成官方 version ID；然后再用 current-head GitHub Actions rerun 收口。

2. Keep `TEST-OPS-001` fail-closed until the repaired current head has an attested rerun
   继续把 `TEST-OPS-001` 视为未完成，不把本地 probe/suite/evidence wiring 写成 non-local pass。

3. Narrow the current Phase 08 / L1 boundary by decision
   通过新的 `DEC-ID` 把 non-local rollout-skew gate 移出当前 release scope，并同步更新 `43`,`44`,`TODO.md` 与 evidence gate。

## Current Recommendation

当前建议保持本问题 open，并采用 Option 1：

* Cloudflare 官方 gradual deployment 语义所暴露的 contract 问题已经回写到 `42` / workflow / tooling 中；当前 remaining gap 已不是“继续猜 probe username / retry 逻辑”，而是要把 runtime observation contract 修正到与 Cloudflare docs 一致，并在 current head 上拿到新的 attested rerun；
* 在 attested run 出来前，`TEST-OPS-001` / `EVID-OPS-001` 仍必须 fail-closed；
* 不要把 `/_ops/v1/healthz`、单 deployment smoke、active deployment identity revalidation，或 `baseline=100%` / `candidate=0%` 下仅靠 version override 命中的 candidate Worker 误写成 skew proof；
* 也不要再把 repo-local `WORKER_VERSION_ID` fallback 当作 attested Cloudflare version ID。

## Closure Criteria

本问题关闭前，必须至少满足以下条件之一：

* 仓库已通过 Spec 明确 dual-version rollout contract，并在 GitHub Actions 中完成可审计的 pre-release skew run，真实证明 `new Worker -> old DO` 与 `old Worker -> new DO`；
* `TEST-OPS-001` / `EVID-OPS-001` 已由新的 attested run 诚实转为 `pass`；
* 或项目通过新的 `DEC-ID` 明确缩减当前 release boundary，并同步更新 `42`,`43`,`44`,`TODO.md` 与 evidence gate。
