## Question

`TEST-OPS-001` 要求 pre-release 非本地门禁真实证明 `new Worker -> old DO` 与 `old Worker -> new DO` 兼容性。`2026-04-03` 继续复核官方 Cloudflare gradual deployment docs 与 current-head remote truth 后，发现这里还不能只表述成“缺 attested rerun”：当前 repo 已把 `42/43/44/23/24/25/26` 中的大部分 dual-version orchestration、pairing observability 与 attested payload contract 落地到 workflow/tooling/route/suite/evidence mapping，并已完成本地 fail-closed 验证；但 official docs 也同时证明，当前 Spec / tooling 曾接受过的一个关键前提是错的，因此本问题现在是“需要先修正 gradual-deployment contract 与实现，再拿 current-head attested pre-release closure”。

截至 `2026-04-02`，官方 Cloudflare 文档与 pinned snapshots 已经给出了四条关键事实：

1. [cloudflare-workers-versions-deployments.md](/root/Matrix/research/sources/cloudflare-workers-versions-deployments.md) 说明 Worker versions 与 deployments 分离，单个 deployment 可同时承载 `1` 或 `2` 个 versions，且 `wrangler versions upload` 不能携带 Durable Object migrations。
2. [cloudflare-workers-gradual-deployments.md](/root/Matrix/research/sources/cloudflare-workers-gradual-deployments.md) 说明可以通过 `Cloudflare-Workers-Version-Key` 与 `Cloudflare-Workers-Version-Overrides` 对 gradual deployment 做稳定路由或显式 version targeting，并明确写出：在 gradual deployment 下，每个 Durable Object 会在当前 deployment 中被分配到某个 Worker version，且“对同一 deployment，同一个 Durable Object 的请求总是落到同一 Worker version”。
3. [cloudflare-do-known-issues.md](/root/Matrix/research/sources/cloudflare-do-known-issues.md) 继续说明普通代码更新本身就可能在短时间内出现 `new Worker -> old DO` 的最终一致窗口，而 gradual deployment 会把这种多版本并存窗口延长到 rollout 期间。
4. 由第 2 条与同页对 `Cloudflare-Workers-Version-Overrides` / `Cloudflare-Workers-Version-Key` 的说明联立可得：这些 header 只控制请求命中哪个 Worker version，并不会改写当前 deployment 已经为某个 Durable Object 决定的 version assignment。因此，`baseline=100%`、`candidate=0%` 再配合 version override，最多只能证明 candidate Worker 可被请求命中，不能诚实地产生 candidate-side DO assignment。

当前 remaining gap 已从“没有 contract”演变为“两件事都还没闭合：一是先修正当前 Spec / tooling 里关于 DO assignment 的错误前提，二是再拿到 current-head attested rerun”：

* `2026-04-02` 进一步核对官方 docs 后，`version_metadata` binding 只提供 version ID/tag/timestamp，不提供 deployment ID；因此 `IF-OPS-009` 现已收敛为由 GitHub Actions harness 显式传入 `dual_version_deployment_id`，后续实现不得再尝试让 worker 在无 account API token 的情况下“猜出”该值。
* 当前 worktree 已把 `IF-OPS-009` / `DATA-OPS-012` / `FLOW-OPS-ROLLOUT-SKEW` 接通为可执行 probe path，并补上 dedicated canonical `tests/pre-release/test-ops-001.test.mjs` 与 `packages/testing/src/evidence.mjs` 的 pre-release mapping；最新本地验证 `npm run governance:check`、`node --test tests/local/control-plane/phase-08-ops.test.mjs`、`node --test tests/local/runtime-foundations/nonlocal-tooling.test.mjs`、`node --test tests/local/runtime-foundations/testing-harness.test.mjs`、`npm run test:all` 均已通过，本地 `npm run evidence:l1 -- --timestamp 20260403T044034Z` 也继续只让 `EVID-GOV-001` 为 `pass`，未出现 fail-open。
* 但当前 repo 仍带着一个已被官方 docs 证伪的 rollout 假设：`42` 与 `packages/testing/src/nonlocal.mjs` 曾允许在 dual-version deployment 中使用 `baseline=100%`、`candidate=0%`，然后依赖 explicit version override 去“制造” candidate-side probe identities。官方 gradual deployment docs 已说明这在带 DO 的场景下不成立，因为 DO assignment 取决于 deployment percentages，而不是 request override。
* pushed-head GitHub Actions rerun `23930798915`（head `b491b55c48e3865ba7bbf375343f5caa24276f0b`）先证明这里剩下的是 current-head execution bug，而不是 contract 缺口：pre-release `Start pre-release rollout skew deployment` 先因 `sanitizeFileName is not defined` 崩溃，workflow `Bundle raw non-local artifacts` 也因把 `run-bundle.tgz` 写回源目录而触发 `tar: file changed as we read it`。这两项缺陷现已在当前未推送 repair worktree 中修复，并补上本地回归。
* pushed-head rerun `23931328404`（head `e833ca13d5b35e7eebd2916b20c2492ab197cd91`）先把 blocker 收敛到 suite 前 deployment identity revalidation：pre-release dual-version start/restore 已成功，但 validation 仍按 baseline gateway deployment id 回读，未跟随 rollout sidecar 中的 `dual_version_deployment_id`，因此 pre-release attestation 继续被 fail-closed 挡住；`ci-integration` / `staging` raw logs 还分别记录了 `500 !== 200` 与 `429 !== 401`。后两项更符合共享 remote 环境上多 test-file 并发执行导致的跨套件干扰；这是基于 raw logs 的工程推断，而不是新的 attested pass/fail 结论。当前未推送 repair worktree 已据此先补一轮修复：pre-release rollout 场景现在会按 `rolloutState.dual_version_deployment_id` 做 suite 前 Cloudflare revalidation，并把 node runner 固定为 `--test-concurrency=1`。
* 更近一轮 latest pushed-head rerun `23932130160`（head `21b1e59baf78c7de7410b97baa76940d42e038c0`）进一步证明 blocker 仍在 current-head harness execution，而不是 `TEST-OPS-001` contract / canonical mapping：dual-version start/restore 继续成功，但 pre-release attestation 仍未生成；复核 raw artifacts 后可确认，这轮真正未修到的点是 `runEnvironmentBackedSuite()` 的 `before_readiness` 仍沿用 baseline deployment summary 做 Cloudflare deployment revalidation，没有像 `before_suite` 那样切到 `rolloutState.dual_version_deployment_id`。同一 run 中 `staging` 还在 `TEST-DER-001` 的 derived search eventual-consistency assertion 上失败，这说明 current-head rerun 也同时受 DER 误报风险影响；但这不是 `TEST-OPS-001` contract 本身的新缺口。当前未推送 repair worktree 已据此再补一轮修复：pre-release rollout 场景现在会在 `before_readiness` 与 `before_suite` 两次 validation 都使用 rollout-scoped deployment id。
* `2026-04-03` 的 current-head rerun `23932691478` 又把这条前提错误从“文档推理”变成了“remote failure truth”：该 run 的 `nonlocal (pre-release)` 已成功完成 dual-version start、suite execution 与 restore，但 `tests/pre-release/test-ops-001.test.mjs` 在真实 probe 请求上返回 `409 !== 200`，raw `pre-release.json` 进一步记录 `error_message = "Suite artifact parsing failed: rollout_skew_probe sidecar is required when rolloutState is provided"`，说明 probe 自身就没有产出 `rollout_skew_probe`。在这次 run 中，`before_readiness` / `before_suite` 都已经对着 `dual_version_deployment_id` 做 validation，因此 remaining blocker 已不再是 deployment revalidation，而是 rollout probe 仍基于错误的 `100/0 + override` / “seed 一次就得到目标 DO version” 假设。
* 因此，在新的 pushed-head GitHub Actions run 成功完成“修正后的 dual-version deployment composition + bounded seed sampling + `TEST-OPS-001` suite + attestation generation + `evidence-l1` 消费”之前，`EVID-OPS-001` 仍不能被诚实写成 `pass`。

因此，`TEST-OPS-001` 不再能被描述成“只差 attested rerun”的状态：当前 repo 已具备大部分 `SR3` 资产，但仍必须先修复 official-doc-backed rollout contract 和对应实现，然后才能进入真正的 attested non-local closure。

## Why It Matters

* [42-deployment-migration-and-recovery.md](/root/Matrix/spec/framework/42-deployment-migration-and-recovery.md) 已把 `REQ-OPS-010`,`REQ-OPS-011`,`REQ-OPS-012` 固定为 rollout compatibility gate，而不是 deployment smoke。

* [44-verification-and-evidence-register.md](/root/Matrix/spec/framework/44-verification-and-evidence-register.md) 要求 `EVID-OPS-001` 是 rollout compatibility attested pre-release report；当前若不把 contract gap 显式化，后续很容易继续把 placeholder suite 写成“已有门禁”。

## Options

1. Repair the rollout contract and implementation, then rerun attested pre-release proof
   先把 `100/0 + override` 这条错误前提从 Spec / workflow / tooling 中清除，改成符合官方 docs 的 dual-version deployment + bounded sampling，再用 current-head GitHub Actions rerun 收口。

2. Keep `TEST-OPS-001` fail-closed until the repaired current head has an attested rerun
   继续把 `TEST-OPS-001` 视为未完成，不把本地 probe/suite/evidence wiring 写成 non-local pass。

3. Narrow the current Phase 08 / L1 boundary by decision
   通过新的 `DEC-ID` 把 non-local rollout-skew gate 移出当前 release scope，并同步更新 `43`,`44`,`TODO.md` 与 evidence gate。

## Current Recommendation

当前建议保持本问题 open，并采用 Option 1：

* contract、workflow、tooling、route、suite 与 evidence mapping 虽已大体进入当前 worktree，但 Cloudflare 官方 gradual deployment 语义已经证明其中一个前提错误；后续必须先修正 `42` / workflow / rollout probe 实现，再执行 current-head GitHub Actions attested rerun；
* 在 attested run 出来前，`TEST-OPS-001` / `EVID-OPS-001` 仍必须 fail-closed；
* 不要把 `/_ops/v1/healthz`、单 deployment smoke、active deployment identity revalidation，或 `baseline=100%` / `candidate=0%` 下仅靠 version override 命中的 candidate Worker 误写成 skew proof。
