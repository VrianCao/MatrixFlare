## Question

`TEST-OPS-001` 要求 pre-release 非本地门禁真实证明 `new Worker -> old DO` 与 `old Worker -> new DO` 兼容性。`2026-04-03` 复核时，`42/43/44/23/24/25/26` 中的 dual-version orchestration、pairing observability 与 attested payload contract 已落地到当前 worktree 的 workflow/tooling/route/suite/evidence mapping，并已完成本地 fail-closed 验证；但当前代码状态仍未拿到新的 GitHub Actions attested pass，因此本问题已从“缺 contract / 缺实现”收敛为“缺 current-head attested pre-release closure”。

截至 `2026-04-02`，官方 Cloudflare 文档与 pinned snapshots 已经给出了三条关键事实：

1. [cloudflare-workers-versions-deployments.md](/root/Matrix/research/sources/cloudflare-workers-versions-deployments.md) 说明 Worker versions 与 deployments 分离，单个 deployment 可同时承载 `1` 或 `2` 个 versions，且 `wrangler versions upload` 不能携带 Durable Object migrations。
2. [cloudflare-workers-gradual-deployments.md](/root/Matrix/research/sources/cloudflare-workers-gradual-deployments.md) 说明可以通过 `Cloudflare-Workers-Version-Key` 与 `Cloudflare-Workers-Version-Overrides` 对 gradual deployment 做稳定路由或显式 version targeting，并明确写出：在 gradual deployment 下，每个 Durable Object 会在当前 deployment 中被分配到某个 Worker version，且“对同一 deployment，同一个 Durable Object 的请求总是落到同一 Worker version”；这使 Worker/DO 的跨版本交互更可能发生，而不是更少。
3. [cloudflare-do-known-issues.md](/root/Matrix/research/sources/cloudflare-do-known-issues.md) 继续说明普通代码更新本身就可能在短时间内出现 `new Worker -> old DO` 的最终一致窗口，而 gradual deployment 会把这种多版本并存窗口延长到 rollout 期间。

当前 remaining gap 已从“没有 contract”收敛为“实现已落地、本地验证已通过，但 attested non-local rerun 还没闭合”：

* `2026-04-02` 进一步核对官方 docs 后，`version_metadata` binding 只提供 version ID/tag/timestamp，不提供 deployment ID；因此 `IF-OPS-009` 现已收敛为由 GitHub Actions harness 显式传入 `dual_version_deployment_id`，后续实现不得再尝试让 worker 在无 account API token 的情况下“猜出”该值。
* 当前 worktree 已把 `IF-OPS-009` / `DATA-OPS-012` / `FLOW-OPS-ROLLOUT-SKEW` 接通为可执行 probe path，并补上 dedicated canonical `tests/pre-release/test-ops-001.test.mjs` 与 `packages/testing/src/evidence.mjs` 的 pre-release mapping；`npm run governance:check`、`node --test tests/local/runtime-foundations/testing-harness.test.mjs`、`npm run test:all` 均已通过，本地 `npm run evidence:l1 -- --timestamp 20260403T012436Z` 也继续只让 `EVID-GOV-001` 为 `pass`，未出现 fail-open。
* 但这些结果仍是本地 truth，不是 non-local attestation。直到新的 pushed-head GitHub Actions run 成功完成 pre-release dual-version start/restore、`TEST-OPS-001` suite、attestation generation 与 `evidence-l1` 消费前，`EVID-OPS-001` 仍不能被诚实写成 `pass`。

因此，`TEST-OPS-001` 已达到 `SR3-ready` 的实现/本地验证状态，但尚未达到 attested non-local 完成态。

## Why It Matters

* [42-deployment-migration-and-recovery.md](/root/Matrix/spec/framework/42-deployment-migration-and-recovery.md) 已把 `REQ-OPS-010`,`REQ-OPS-011`,`REQ-OPS-012` 固定为 rollout compatibility gate，而不是 deployment smoke。
* [43-testing-and-compliance.md](/root/Matrix/spec/framework/43-testing-and-compliance.md) 把 `TEST-OPS-001` 定义为 pre-release 非本地门禁；如果没有真实 dual-version orchestration，就不能再把 “healthz 正常” 或 “active deployment identity 已回读” 误写成 version skew proof。
* [44-verification-and-evidence-register.md](/root/Matrix/spec/framework/44-verification-and-evidence-register.md) 要求 `EVID-OPS-001` 是 rollout compatibility attested pre-release report；当前若不把 contract gap 显式化，后续很容易继续把 placeholder suite 写成“已有门禁”。

## Options

1. Keep `TEST-OPS-001` fail-closed until a current-head attested rerun exists
   继续把 `TEST-OPS-001` 视为未完成，不把本地 probe/suite/evidence wiring 写成 non-local pass。

2. Execute the already-specified gradual-deployment orchestration on current head
   既有 Spec / workflow / tooling / route / suite 已落地后，继续以 GitHub Actions current-head attested rerun 收口，不再把剩余工作误写成“继续定义或实现 contract”。

3. Narrow the current Phase 08 / L1 boundary by decision
   通过新的 `DEC-ID` 把 non-local rollout-skew gate 移出当前 release scope，并同步更新 `43`,`44`,`TODO.md` 与 evidence gate。

## Current Recommendation

当前建议保持本问题 open，并采用 Option 2：

* contract、workflow、tooling、route、suite 与 evidence mapping 已进入当前 worktree，后续应直接执行 current-head GitHub Actions attested rerun，而不是继续把 blocker 表述成“还缺实现”；

* 不要把 `/_ops/v1/healthz`、单 deployment smoke、或 active deployment identity revalidation 误写成 skew proof。
