## Question

`TEST-OPS-001` 要求 pre-release 非本地门禁真实证明 `new Worker -> old DO` 与 `old Worker -> new DO` 兼容性，但当前仓库还没有一个 Spec-ready、attestable 的 gradual deployment orchestration contract。

截至 `2026-04-02`，官方 Cloudflare 文档与 pinned snapshots 已经给出了三条关键事实：

1. [cloudflare-workers-versions-deployments.md](/root/Matrix/research/sources/cloudflare-workers-versions-deployments.md) 说明 Worker versions 与 deployments 分离，单个 deployment 可同时承载 `1` 或 `2` 个 versions，且 `wrangler versions upload` 不能携带 Durable Object migrations。
2. [cloudflare-workers-gradual-deployments.md](/root/Matrix/research/sources/cloudflare-workers-gradual-deployments.md) 说明可以通过 `Cloudflare-Workers-Version-Key` 与 `Cloudflare-Workers-Version-Overrides` 对 gradual deployment 做稳定路由或显式 version targeting，并明确写出：在 gradual deployment 下，每个 Durable Object 会在当前 deployment 中被分配到某个 Worker version，且“对同一 deployment，同一个 Durable Object 的请求总是落到同一 Worker version”；这使 Worker/DO 的跨版本交互更可能发生，而不是更少。
3. [cloudflare-do-known-issues.md](/root/Matrix/research/sources/cloudflare-do-known-issues.md) 继续说明普通代码更新本身就可能在短时间内出现 `new Worker -> old DO` 的最终一致窗口，而 gradual deployment 会把这种多版本并存窗口延长到 rollout 期间。

但当前 repo truth 仍未闭合为可执行 contract：

* `.github/workflows/nonlocal-phase08.yml` 与 `packages/testing/src/nonlocal.mjs` 目前只实现了单次 `wrangler deploy` 的 all-at-once non-local deploy/readiness/attestation 链路，没有 dual-version pre-release rollout orchestration。
* 仓库没有一个被 `43/44/26` 固化的 release-gate contract，说明 pre-release harness 应如何：
  * 上传两个明确可区分的 Worker versions；
  * 创建包含双版本的 active deployment；
  * 对 gateway/ops surface 施加可信 version override 或 version key；
  * 记录“请求命中了哪一个 Worker version”以及“状态访问实际命中了哪一个 DO version”的 attested 观测。
* 当前公开 / 控制面 HTTP surface 最多只能稳定回读当前 Worker 的 `worker_version_id` / `deployment_composition`；它还不能把一次 stateful request 的 Worker-version/DO-version pairing 作为可审计证据导出。因此，即使仓库今天补一个 `tests/pre-release/test-ops-001.test.mjs` 文件，也仍会缺少 honest proof 所需的 orchestration 与 observability contract。

因此，`TEST-OPS-001` 对当前 repo 而言仍低于 `SR2`：平台候选能力存在，但 release-gate contract 与 attested proof path 尚未被规范收敛。

## Why It Matters

* [42-deployment-migration-and-recovery.md](/root/Matrix/spec/framework/42-deployment-migration-and-recovery.md) 已把 `REQ-OPS-010`,`REQ-OPS-011`,`REQ-OPS-012` 固定为 rollout compatibility gate，而不是 deployment smoke。
* [43-testing-and-compliance.md](/root/Matrix/spec/framework/43-testing-and-compliance.md) 把 `TEST-OPS-001` 定义为 pre-release 非本地门禁；如果没有真实 dual-version orchestration，就不能再把 “healthz 正常” 或 “active deployment identity 已回读” 误写成 version skew proof。
* [44-verification-and-evidence-register.md](/root/Matrix/spec/framework/44-verification-and-evidence-register.md) 要求 `EVID-OPS-001` 是 rollout compatibility attested pre-release report；当前若不把 contract gap 显式化，后续很容易继续把 placeholder suite 写成“已有门禁”。

## Options

1. Keep `TEST-OPS-001` fail-closed until a gradual-deployment contract exists
   继续把 `TEST-OPS-001` 视为未完成，不用 placeholder suite 假装 coverage 已具备。

2. Define and implement a gradual-deployment orchestration contract
   在 Spec 中明确双版本 pre-release rollout、version override/version key、观测与 attestation contract，然后在 GitHub Actions 中真实执行。

3. Narrow the current Phase 08 / L1 boundary by decision
   通过新的 `DEC-ID` 把 non-local rollout-skew gate 移出当前 release scope，并同步更新 `43`,`44`,`TODO.md` 与 evidence gate。

## Current Recommendation

当前建议保持本问题 open，并采用 Option 1 直到 Option 2 真正落地：

* 继续把 `TEST-OPS-001` / `EVID-OPS-001` 维持 fail-closed；
* 不要把 `/_ops/v1/healthz`、单 deployment smoke、或 active deployment identity revalidation 误写成 skew proof；
* 若项目要在 Phase 08 内闭合本项，必须先把 gradual deployment orchestration + observability + attestation contract 写进 Spec，再实现 workflow/tooling。

## Closure Criteria

本问题关闭前，必须至少满足以下条件之一：

* 仓库已通过 Spec 明确 dual-version rollout contract，并在 GitHub Actions 中完成可审计的 pre-release skew run，真实证明 `new Worker -> old DO` 与 `old Worker -> new DO`；
* `TEST-OPS-001` / `EVID-OPS-001` 已由新的 attested run 诚实转为 `pass`；
* 或项目通过新的 `DEC-ID` 明确缩减当前 release boundary，并同步更新 `42`,`43`,`44`,`TODO.md` 与 evidence gate。
