## Question

截至 `2026-04-05` 当前 pushed head `0a7a8c915e84514b55f28cc01f628933467af5d6`，仓库已经真实包含 `prod-install` / `release-candidate` / `promote-prod` / `rollback-prod` / `prod-cost-monthly` 五条 production automation workflow 与对应 CLI/tooling/local tests；本地 gate 也已重新执行并保持 fail-closed：

* `npm run governance:check`
* `node --test tests/local/runtime-foundations/nonlocal-tooling.test.mjs`
* `node --test tests/local/runtime-foundations/testing-harness.test.mjs`
* `npm run test:all`
* `npm run evidence:l1 -- --timestamp 20260405T084741Z`

但 `08.05A` 所要求的 same-head remote proof 目前被一个新的官方 GitHub Actions 平台约束卡住，而不是被本地 contract/test 缺口卡住。

本轮已得到两条可审计事实：

1. 仓库默认分支当前仍是 `master`（`gh repo view VrianCao/MatrixFlare --json defaultBranchRef` 返回 `master`）。
2. 对当前分支 `phase08-nonlocal-closure` 执行：
   `gh workflow run prod-install.yml --repo VrianCao/MatrixFlare --ref phase08-nonlocal-closure -f release_ref=0a7a8c915e84514b55f28cc01f628933467af5d6`
   得到 GitHub API `404`：
   `workflow prod-install.yml not found on the default branch`

这与 GitHub 官方文档的当前行为一致：`workflow_dispatch` 手动触发前提是 workflow 文件已存在于 repository default branch。换言之，在这些 production automation workflow 仍只存在于 feature branch、尚未进入 default branch `master` 的当前状态下，项目无法为当前 head 直接获得 `prod-install` / `release-candidate` / `promote-prod` / `rollback-prod` / `prod-cost-monthly` 的 same-head remote proof。

因此，当前 active blocker 已经从“是否有 workflow/tooling”收窄为更精确的 truth：

* workflow/tooling/local tests 已存在；
* current-head push 已存在；
* 但 GitHub 官方 `workflow_dispatch` 默认分支限制使 newly-added prod workflows 在 feature branch 上根本无法被手动 dispatch；
* 所以 `08.05A` / `TEST-OPS-003` / `EVID-OPS-003` 当前仍不能诚实宣称拿到了 same-head remote proof；
* 这不是 Cloudflare topology 或 Billing scope 本身的 blocker，而是 GitHub Actions 入口可用性的 blocker。

## Why It Matters

* [42-deployment-migration-and-recovery.md](/root/Matrix/spec/framework/42-deployment-migration-and-recovery.md) 已把 GitHub Actions 固定为 production automation 的执行入口；若忽略 `workflow_dispatch` 的默认分支限制，就会把“无法启动 workflow”误写成“workflow 尚未验证”或更糟的“已经完成”。
* [DEC-0005](/root/Matrix/spec/decisions/DEC-0005.md) 已把 `prod-install` / `release-candidate` / `promote-prod` / `rollback-prod` / `prod-cost-monthly` 固定为默认生产路径；如果这些 workflow 还不能在当前 head 上被 GitHub 真正 dispatch，就不存在 same-head remote proof。
* [TODO.md](/root/Matrix/TODO.md) 的 `08.05A` 当前完成标准明确要求 same-head GitHub Actions truth 或 blocker 写回正确 artifact；这个问题若不记录，就会逼团队在没有 remote run 的情况下假装已经“只差 Cloudflare”。

## Options

1. Bootstrap the workflow definitions onto the default branch first
   先让这些 workflow 文件进入 `master`，然后再从 current head 重新触发 same-head remote proof。

2. Introduce a default-branch proxy workflow by decision
   通过新的 `DEC-ID` 固定一条已在 default branch 存在的 dispatcher / reusable workflow，由它去调用 feature-branch 当前 head 的 prod automation 逻辑。

3. Keep the blocker open and do not overclaim same-head proof
   在 workflow definitions 进入 default branch 或有新的 dispatcher contract 前，继续把 `08.05A` / `EVID-OPS-003` 保持未完成。

## Current Recommendation

当前建议保持本问题 open，并采用 Option 3：

* 不把“workflow 已写出来”误写成“same-head remote proof 已可执行”；
* 在 `TODO.md` / control spec / source observation register 中明确写出 default-branch blocker；
* 只有当这些 workflow definitions 已进入 default branch，或项目通过新的 `DEC-ID` 固定 default-branch proxy path 后，才重新尝试 same-head remote proof。

## Closure Criteria

本问题关闭前，必须至少满足以下条件之一：

* 这些 production automation workflow 已真实存在于 repository default branch，并已对 current head 触发 same-head remote runs；
* 项目通过新的 `DEC-ID` 固定一个官方、可审计的 default-branch dispatcher / reusable-workflow contract，并已据此拿到 same-head remote proof；
* 或 GitHub 官方文档/能力变化，使 workflow_dispatch 不再要求 workflow 文件位于 default branch，且该变化已被纳入 spec stack 与 remote truth。
