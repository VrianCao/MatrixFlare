## Question

Phase 08 当前的 non-local 拓扑只会给三套环境部署 `*.workers.dev` Worker URL，并把 `ops-worker` 的 `ACCESS_TEAM_DOMAIN` 写成 `<environment>.cloudflareaccess.invalid` 占位值；仓库内既没有已验证的 active-zone custom management hostname，也没有真正启用在 account-owned `workers.dev`/Preview URL 上的 Access ingress，更没有真实 Access application、team domain、AUD、或 GitHub Actions 可用的 Access service token/JWT contract。

这与当前规范和 Cloudflare 官方事实之间出现了未决缝隙：

* 规范要求 `ops-worker` 只能通过 Access 保护的专用 management ingress 进入，并且应用层只信任 `Cf-Access-Jwt-Assertion`，不直接信任 service-token headers。见 `CF-NET-004`,`CF-NET-005`,`CF-NET-006`,`CF-NET-007`,`DATA-D1-006`,`IF-OPS-001`-`008`,`REQ-OPS-010`-`012`。
* Cloudflare 官方 Access 文档与本地快照 [cloudflare-access-applications.md](/root/Matrix/research/sources/cloudflare-access-applications.md) 仍要求 self-hosted public application 的 domain 属于当前账户 active zone；而独立的 Workers 官方文档与本地快照 [cloudflare-workers-dev.md](/root/Matrix/research/sources/cloudflare-workers-dev.md) 又明确写明 account-owned `workers.dev` 与 Preview URL 可以直接启用 Cloudflare Access。
* 这说明“必须 active-zone custom hostname 才能有合法 Access ingress”并不是当前官方语义；真正未解决的是：仓库到底采用哪条官方支持路径，以及如何把 team domain、AUD、service-auth 和 GitHub Actions automation contract 落到可审计工件里。
* 当前 non-local harness 使用的却是公开 `workers.dev` URL 和无效 team-domain placeholder，因此即使 workflow 已能部署、跑 smoke、上传 raw bundle、生成 attestation，也仍无法诚实产出“通过 Cloudflare Access 保护后再调用 `/_ops`”的正向证明链。

因此，以下问题目前都还没有被规范收敛：

* Phase 08 是否应优先采用 “启用在 account-owned `workers.dev`/Preview URL 上的 Access” 还是 “active-zone custom management hostname” 作为 `/_ops` 的合法非本地 ingress；
* 若采用前者，GitHub Actions 是否可以用当前 Cloudflare 官方支持的 surface 自动化启用/维护 `workers.dev` Access，并拿到真实 team domain、AUD 与 service-auth 输入；
* 若采用后者，当前仓库允许依赖的最小输入集是什么：zone/domain、Access app 配置、service token、还是更高权限的 Cloudflare API token；
* GitHub Actions 是否可以用现有 Cloudflare API 自动化创建/维护所需 Access application 与 service token，还是仍必须依赖额外人工预配；
* 在上述问题未解决前，`TEST-DER-001` / `EVID-DER-001` 与 `TEST-OPS-001` / `EVID-OPS-001` 是否只能继续 fail-closed，而不能被描述成“只差 dedicated suite 文件名”。

## Why It Matters

* [21-runtime-topology-and-platform-model.md](/root/Matrix/spec/framework/21-runtime-topology-and-platform-model.md) 与 [22-data-consistency-and-routing.md](/root/Matrix/spec/framework/22-data-consistency-and-routing.md) 已把 `/_ops/*` 固定为 Access-protected management ingress，而不是普通公网 surface。
* [40-security-and-abuse-resistance.md](/root/Matrix/spec/framework/40-security-and-abuse-resistance.md) 与 [23-interface-contract-catalog.md](/root/Matrix/spec/framework/23-interface-contract-catalog.md) 要求 `ops-worker` 只信任经 Access 生成的 JWT；若 non-local harness 实际没有 Access 入口，则 `TEST-DER-001` rebuild 与 `TEST-OPS-001` compatibility gate 根本没有合法正向入口。
* `TODO.md` 的 `08.04` / `08.05` 目前仍把 `TEST-DER-001` / `TEST-OPS-001` 标为未完成；如果不把这个拓扑前置条件显式化，后续很容易继续误把问题描述成“只缺 suite/mapping”，从而夸大完成度。

## Options

1. Keep `workers.dev`-only non-local topology and accept no positive ops-authenticated proof in Phase 08
   继续只在 `workers.dev` 上部署 non-local 环境，把 `TEST-DER-001` rebuild consistency 与 `TEST-OPS-001` version-skew 维持 fail-closed，不尝试闭合 Access-protected `/_ops` 正向路径。

2. Enable Access on account-owned `workers.dev` or Preview URLs and automate service-auth for GitHub Actions
   保持当前 non-local URL 形态，但把 Access 真正启用到 account-owned `workers.dev`/Preview URL 上，并补齐 team domain、AUD、service token / policy 与 workflow automation，使请求到达 `ops-worker` 时表现为可验证 JWT。

3. Introduce a real management hostname on an active Cloudflare zone and automate Access service-auth for GitHub Actions
   为 `ops-worker` 引入真实 management domain、Access application、service token / policy，并让 workflow 在 Access 边缘完成鉴权，到达 `ops-worker` 时表现为可验证 JWT。

4. Narrow the product/test boundary by decision
   通过新的 `DEC-ID` 正式把 `TEST-DER-001` rebuild consistency 或 `TEST-OPS-001` non-local gate 从当前 Phase 08/L1 目标中移出，明确承认当前产品边界不提供这类 positive non-local proof。

## Current Recommendation

当前建议保持本问题 open，并把 Option 2/3 都视为官方允许的候选方向、Option 1 作为当前诚实状态：

* 在真实 Access ingress 与 automation contract 真正落地前，不得宣称 `TEST-DER-001` 或 `TEST-OPS-001` 只是“还没补 dedicated suite”；
* `TODO.md` 必须把 “non-local ops Access topology” 记为 `08.04` / `08.05` 的前置条件之一；
* 在 GitHub Actions 仍只持有 `CLOUDFLARE_ACCOUNT_ID` / `CLOUDFLARE_API_TOKEN`、且没有证明能自动获得真实 Access ingress 的情况下，继续让 `EVID-DER-001` / `EVID-OPS-001` fail-closed；
* 若项目希望在 Phase 08 内闭合 `TEST-DER-001` 或 `TEST-OPS-001`，必须先证明以下两者之一：
  account-owned `workers.dev`/Preview URL 的 Access enablement + team domain/AUD/service-auth automation；
  active-zone custom management hostname + Access app/service-auth automation。

## Closure Criteria

本问题关闭前，必须至少满足以下条件之一：

* 仓库已经为 non-local `ops-worker` 配置真实 Access ingress，并满足以下至少一条：
  account-owned `workers.dev`/Preview URL 上的 Access 已启用且可审计；
  active-zone custom management hostname 已配置且可审计；
* GitHub Actions 已能通过官方支持的 Access service-auth 或等价机制，对该 ingress 发起请求，并让 `ops-worker` 接收到可验证的 `Cf-Access-Jwt-Assertion`；
* `TEST-DER-001` 的 rebuild consistency 与 `TEST-OPS-001` 的 version-skew gate 已得到 honest non-local dedicated coverage，并回链到新的 attested run；
* 或者项目通过新的 `DEC-ID` 明确缩减当前 Phase 08 / `L1` 边界，并同步更新 `43`,`44`,`TODO.md` 与 evidence gate。
