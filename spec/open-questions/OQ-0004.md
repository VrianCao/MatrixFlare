## Question

截至当前 worktree，仓库已经不再只有 placeholder Access 叙述。`packages/testing/src/nonlocal.mjs` 与 `.github/workflows/nonlocal-phase08.yml` 现在都包含了一条明确的候选 contract：在 `staging` / `pre-release` 对应的 account-owned `workers.dev` hostname 上 ensure Cloudflare Access application，读取 Zero Trust organization `auth_domain`，创建或轮换稳定 service token，ensure 绑定该 token 的 service-auth policy，远端 bootstrap `DATA-D1-006` 所需的 control-plane D1 schema / operator policy，再把 `service_token_client_id` / `service_token_client_secret` 传入 non-local test harness，由 `tests/staging/*` / `tests/pre-release/*` 对真实受保护的 `/_ops` surface 发起请求。

`2026-04-02T19:47:44Z` 完成的 GitHub Actions run `23918727505`（code state `7bfb527e28dba368e7b9622b9078e43469182000`）已经把这条 contract 的 live status 从“尚未证明”推进为“Access ingress 已被真实证明可工作，但 current-head attested closure 仍未完成”：

* `staging` / `pre-release` 的 `Prepare non-local ops Access session` 都成功；
* 两环境都先通过了 Access-protected `/_ops/v1/healthz`；
* 失败点不再是 `401` / placeholder domain / HTML `500`，而是 `ops_rebuild_start` 对 `/_ops/v1/rebuilds` 返回 `503` + `D1_EXEC_ERROR: ... BEGIN IMMEDIATE; ... use state.storage.transaction()`；
* 这说明 workers.dev Access path、organization lookup、application ensure、policy ensure、service token materialization 与 live `Cf-Access-Jwt-Assertion` ingress 已经不再是 active blocker；当前未闭环的是 `ops-worker` 进入 rebuild 正向路径后，control-plane D1 写事务仍不兼容 Cloudflare managed D1 的运行时语义。

当前未决点因此变成：

* 规范要求 `ops-worker` 只能通过 Access 保护的专用 management ingress 进入，并且应用层只信任 `Cf-Access-Jwt-Assertion`，不直接信任 service-token headers。见 `CF-NET-004`,`CF-NET-005`,`CF-NET-006`,`CF-NET-007`,`DATA-D1-006`,`IF-OPS-001`-`008`,`REQ-OPS-010`-`012`。
* Cloudflare 官方 Access 文档与本地快照 [cloudflare-access-applications.md](/root/Matrix/research/sources/cloudflare-access-applications.md) 仍要求 self-hosted public application 的 domain 属于当前账户 active zone；而独立的 Workers 官方文档与本地快照 [cloudflare-workers-dev.md](/root/Matrix/research/sources/cloudflare-workers-dev.md) 又明确写明 account-owned `workers.dev` 与 Preview URL 可以直接启用 Cloudflare Access。
* 这说明“必须 active-zone custom hostname 才能有合法 Access ingress”并不是当前官方语义；当前 repo 已实际选择先落地 workers.dev Access 路径，并把 team domain、AUD、service-auth 与 GitHub Actions automation contract 写进了 non-local tooling。
* 本轮本地测试还确认了一个此前会破坏该路径的真实 bug：`ops-worker` 的 Access JWT verifier 曾错误要求 `sub` 必须非空；在 service-auth JWT 允许空 `sub` 的官方语义下，这会让 live workers.dev Access 调用在到达 origin 后误判为 unauthorized。该问题现已修复并被 local control-plane tests 锁定。
* workflow/tooling 侧原先允许 staging / pre-release 在缺失 `ops-access-session` 时继续尝试 `/_ops` 非本地门禁的 bypass 也已被移除：`.github/workflows/nonlocal-phase08.yml` 不再容忍 `nonlocal-prepare-ops-access` 失败，`packages/testing/src/nonlocal.mjs` 现在会在 readiness 与 suite execution 前强制要求 `MATRIX_REMOTE_OPS_BASE_URL` 与 `MATRIX_REMOTE_OPS_ACCESS_CLIENT_{ID,SECRET}`。换言之，当前 repo-side contract 已经不仅“能准备 Access session”，而且会在缺失该 session 时 fail-closed。

因此，以下问题目前都还没有被规范收敛：

* repo 当前未提交的 `packages/control-plane/src/persistence.mjs` `batch()` 事务修复，是否足以让 `/_ops/v1/rebuilds` 在真实 Cloudflare managed D1 上通过，而不再触发 raw transaction SQL `D1_EXEC_ERROR`；
* 在 Access ingress 已被 live 证明成立之后，current-head attested rerun 是否还能暴露新的 repo-side blocker，例如 D1 `batch()` 调用形状、控制面读写顺序、或后续 rebuild queue/probe 路径上的其他兼容性问题；
* 在 current-head attested proof 出来前，`TEST-DER-001` 是否仍必须保持 fail-closed 为“repo-side contract 已落地，但 non-local proof 尚未闭环”；
* 即使 workers.dev Access path 闭合，`TEST-OPS-001` 是否仍需继续被 `OQ-0005` 的 rollout-skew contract 单独阻塞，而不能被 `08.04B` 成功替代。

## Why It Matters

* [21-runtime-topology-and-platform-model.md](/root/Matrix/spec/framework/21-runtime-topology-and-platform-model.md) 与 [22-data-consistency-and-routing.md](/root/Matrix/spec/framework/22-data-consistency-and-routing.md) 已把 `/_ops/*` 固定为 Access-protected management ingress，而不是普通公网 surface。
* [40-security-and-abuse-resistance.md](/root/Matrix/spec/framework/40-security-and-abuse-resistance.md) 与 [23-interface-contract-catalog.md](/root/Matrix/spec/framework/23-interface-contract-catalog.md) 要求 `ops-worker` 只信任经 Access 生成的 JWT；run `23918727505` 现在已经给出了一轮 live proof，证明 workers.dev Access ingress 不再是 placeholder，但这还不等于 current-head 的 rebuild/control-plane transaction path 已完成 attested closure。
* `TODO.md` 的 `08.04` / `08.05` 现在必须区分三类 blocker：workers.dev Access current-head proof (`OQ-0004`)、rollout-skew contract (`OQ-0005`) 与 cost proof contract (`OQ-0006`)；若继续把它们混成“只差 suite/mapping”或“还是没 Access automation”，会同时夸大 repo 进度与 blocker 严重度。

## Options

1. Keep the current workers.dev Access automation path and obtain current-head attested proof
   继续沿用当前 repo 已实现的 workers.dev Access candidate contract，但必须通过新的 GitHub Actions attested run 证明其在真实账号上可工作。

2. Abandon workers.dev Access and move to an active-zone management hostname
   为 `ops-worker` 引入真实 management domain、Access application、service token / policy，并让 workflow 在 Access 边缘完成鉴权，到达 `ops-worker` 时表现为可验证 JWT。

3. Narrow the product/test boundary by decision
   通过新的 `DEC-ID` 正式把 `TEST-DER-001` rebuild consistency 或 `TEST-OPS-001` non-local gate 从当前 Phase 08/L1 目标中移出，明确承认当前产品边界不提供这类 positive non-local proof。

## Current Recommendation

当前建议保持本问题 open，并把当前 repo 已实现且已在 live run 中通过 healthz 的 workers.dev Access path 视为唯一主线候选：

* 不再把本项描述成“仓库还没有 Access automation contract”或“workers.dev path 是否可行未知”；当前更准确的表述是 “workers.dev Access ingress 已在 live run 中穿透到 `/_ops`，但 current-head attested rebuild closure 尚未完成”；
* `TODO.md` 必须把 `08.04B` 记为 “代码已落地、待 non-local attested rerun” 的前置条件，而不是继续写成 placeholder team-domain 阶段；
* `staging` / `pre-release` 现在必须先拿到 `ops-access-session` 导出的 `MATRIX_REMOTE_OPS_*` 环境变量，才能进入 readiness 与 suite execution；缺失 session 时应在 test execution 之前直接 fail-closed，而不是退化为公开 `workers.dev` 探测；
* 当前 repo-side repair 主线已经从 `ops-handler` / `services` 的 typed error leak，推进到 `packages/control-plane/src/persistence.mjs` 的 managed-D1 transaction compatibility；在新的 attested rerun 前，不得把这轮 local `batch()` 修复误写成 non-local 已闭环；
* 在 current-head attested run 证明真实 Access ingress 闭合前，继续让 `EVID-DER-001` fail-closed；
* `TEST-SEC-001` 的负向 `401` 只能证明 `/_ops` 入口继续 fail-closed；`08.04B` 的 closure 仍必须依赖成功的 Access-authenticated `/_ops` 请求与 live `Cf-Access-Jwt-Assertion` validation，而不能把负向拒绝断言当成拓扑闭环替代；
* 即使本问题关闭，`TEST-OPS-001` / `EVID-OPS-001` 也仍可能继续被 `OQ-0005` 单独阻塞，不得把 Access 成功误写成 rollout-skew proof 已具备。

## Closure Criteria

本问题关闭前，必须至少满足以下条件之一：

* 来自当前代码状态的 GitHub Actions attested run 已证明 workers.dev Access automation 真正闭合，至少包括：Access org/app/policy/service token provisioning 成功、`/_ops/v1/healthz` / `/_ops/v1/rebuilds` 经 Access 成功、`ops-worker` 接收到可验证 `Cf-Access-Jwt-Assertion`，且不再因 control-plane D1 raw transaction SQL 失败；
* `TEST-DER-001` 已由新的 staging / pre-release attested run 诚实转为 `pass`；
* 或者若 workers.dev path 被证明不可行，项目改走 active-zone management hostname 并通过新的 attested run 完成同等证明；
* 或者项目通过新的 `DEC-ID` 明确缩减当前 Phase 08 / `L1` 边界，并同步更新 `43`,`44`,`TODO.md` 与 evidence gate。
