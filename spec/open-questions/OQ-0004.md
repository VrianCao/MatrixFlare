## Question

Phase 08 当前的 non-local 拓扑只会给三套环境部署 `*.workers.dev` Worker URL，并把 `ops-worker` 的 `ACCESS_TEAM_DOMAIN` 写成 `<environment>.cloudflareaccess.invalid` 占位值；仓库内不存在 active-zone custom hostname、真实 Access application、真实 team domain，GitHub Actions 也没有为受保护 `/_ops` surface 注入可用的 Access service token。

这与当前规范和 Cloudflare 官方事实之间出现了未决缝隙：

* 规范要求 `ops-worker` 只能通过 Access 保护的专用 management domain 进入，并且应用层只信任 `Cf-Access-Jwt-Assertion`，不直接信任 service-token headers。见 `CF-NET-004`,`CF-NET-005`,`CF-NET-006`,`CF-NET-007`,`DATA-D1-006`,`IF-OPS-001`-`008`,`REQ-OPS-010`-`012`。
* Cloudflare 官方 Access 文档与本地快照 [cloudflare-access-applications.md](/root/Matrix/research/sources/cloudflare-access-applications.md) 要求 self-hosted application 的 domain 必须属于当前账户中的 active zone；service token 只是在 Access 边缘鉴权，请求到达 origin/Worker 时仍应表现为 Access 生成的 JWT。
* 当前 non-local harness 使用的却是 `workers.dev` URL 和无效 team-domain placeholder，因此即使 workflow 已能部署、跑 smoke、上传 raw bundle、生成 attestation，也无法诚实产出“通过 Cloudflare Access 保护后再调用 `/_ops`”的正向证明链。

因此，以下问题目前都还没有被规范收敛：

* Phase 08 是否必须引入一个真实 active-zone management hostname，才能让 `TEST-DER-001` 的 rebuild consistency 和 `TEST-OPS-001` 的 version-skew gate 进入 honest non-local coverage；
* 若必须引入该 hostname，当前仓库允许依赖的最小输入集是什么：zone/domain、Access app 配置、service token、还是更高权限的 Cloudflare API token；
* GitHub Actions 是否可以用现有 Cloudflare API 自动化创建/维护所需 Access application 与 service token，还是必须依赖额外人工预配；
* 在上述问题未解决前，`TEST-DER-001` / `EVID-DER-001` 与 `TEST-OPS-001` / `EVID-OPS-001` 是否只能继续 fail-closed，而不能被描述成“只差 dedicated suite 文件名”。

## Why It Matters

* [21-runtime-topology-and-platform-model.md](/root/Matrix/spec/framework/21-runtime-topology-and-platform-model.md) 与 [22-data-consistency-and-routing.md](/root/Matrix/spec/framework/22-data-consistency-and-routing.md) 已把 `/_ops/*` 固定为 Access-protected management domain，而不是普通公网 `workers.dev` surface。
* [40-security-and-abuse-resistance.md](/root/Matrix/spec/framework/40-security-and-abuse-resistance.md) 与 [23-interface-contract-catalog.md](/root/Matrix/spec/framework/23-interface-contract-catalog.md) 要求 `ops-worker` 只信任经 Access 生成的 JWT；若 non-local harness 实际没有 Access 入口，则 `TEST-DER-001` rebuild 与 `TEST-OPS-001` compatibility gate 根本没有合法正向入口。
* `TODO.md` 的 `08.04` / `08.05` 目前仍把 `TEST-DER-001` / `TEST-OPS-001` 标为未完成；如果不把这个拓扑前置条件显式化，后续很容易继续误把问题描述成“只缺 suite/mapping”，从而夸大完成度。

## Options

1. Keep `workers.dev`-only non-local topology and accept no positive ops-authenticated proof in Phase 08
   继续只在 `workers.dev` 上部署 non-local 环境，把 `TEST-DER-001` rebuild consistency 与 `TEST-OPS-001` version-skew 维持 fail-closed，不尝试闭合 Access-protected `/_ops` 正向路径。

2. Introduce a real management hostname on an active Cloudflare zone and automate Access service-auth for GitHub Actions
   为 `ops-worker` 引入真实 management domain、Access application、service token / policy，并让 workflow 在 Access 边缘完成鉴权，到达 `ops-worker` 时表现为可验证 JWT。

3. Narrow the product/test boundary by decision
   通过新的 `DEC-ID` 正式把 `TEST-DER-001` rebuild consistency 或 `TEST-OPS-001` non-local gate 从当前 Phase 08/L1 目标中移出，明确承认当前产品边界不提供这类 positive non-local proof。

## Current Recommendation

当前建议保持本问题 open，并以 Option 2 作为推荐方向、Option 1 作为当前诚实状态：

* 在 active-zone management hostname 与 Access automation 真正落地前，不得宣称 `TEST-DER-001` 或 `TEST-OPS-001` 只是“还没补 dedicated suite”；
* `TODO.md` 必须把 “non-local ops Access topology” 记为 `08.04` / `08.05` 的前置条件之一；
* 在 GitHub Actions 仍只持有 `CLOUDFLARE_ACCOUNT_ID` / `CLOUDFLARE_API_TOKEN`、且没有证明能自动获得真实 Access ingress 的情况下，继续让 `EVID-DER-001` / `EVID-OPS-001` fail-closed；
* 若项目希望在 Phase 08 内闭合 `TEST-DER-001` 或 `TEST-OPS-001`，必须先证明 active-zone custom management hostname、Access app/service-auth automation、以及最小 secret/permission contract。

## Closure Criteria

本问题关闭前，必须至少满足以下条件之一：

* 仓库已经为 non-local `ops-worker` 配置真实 management domain，并证明该 hostname 属于目标账户 active zone；
* GitHub Actions 已能通过官方支持的 Access service-auth 或等价机制，对该 management domain 发起请求，并让 `ops-worker` 接收到可验证的 `Cf-Access-Jwt-Assertion`；
* `TEST-DER-001` 的 rebuild consistency 与 `TEST-OPS-001` 的 version-skew gate 已得到 honest non-local dedicated coverage，并回链到新的 attested run；
* 或者项目通过新的 `DEC-ID` 明确缩减当前 Phase 08 / `L1` 边界，并同步更新 `43`,`44`,`TODO.md` 与 evidence gate。
