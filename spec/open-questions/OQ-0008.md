## Question

Matrix `v1.17` 现已把 application-service legacy client auth 语义写成明确要求，而不是“未实现即可忽略”的历史兼容细节：

* `POST /_matrix/client/*/login` 若使用 `m.login.application_service`，且 server 不支持 Legacy Authentication API，必须返回 `400 M_APPSERVICE_LOGIN_UNSUPPORTED`。
* `POST /_matrix/client/*/register` 即使 server 不支持 Legacy Authentication API，也仍必须允许 application service 创建用户；此时 application service 必须设置 `"inhibit_login": true`，否则必须返回 `400 M_APPSERVICE_LOGIN_UNSUPPORTED`。
* `m.login.application_service` 的 invalid `as_token` wire truth 在当前 pinned sources 中至少对 `/login` 明确存在冲突：application-service API 页面写成 `/register` 或 `/login` 无有效 `as_token` -> `401 M_MISSING_TOKEN` / `401 M_UNKNOWN_TOKEN`，而 client-server API `/login` 页面仍保留 invalid/non-appservice token -> `M_FORBIDDEN` 的旧描述。对 `/register`，当前 pinned client-server page 只明确了 `inhibit_login` / `M_APPSERVICE_LOGIN_UNSUPPORTED` special case，并未给出同等级 invalid-token wording；因此本仓库目前只能把 `/login` invalid-token conflict 视为已证实、把 `/register` invalid-token truth 视为尚未由当前 pinned baseline 唯一收敛。

但截至 `2026-04-06` 当前仓库，相关实现前置并未形成可直接编码的 contract：

1. [`spec/framework/34-search-directory-and-appservices.md`](/root/Matrix/spec/framework/34-search-directory-and-appservices.md) 只定义了 appservice 配置、namespace ownership、txn delivery 与 ops control-plane，不存在“`gateway-worker` 如何在 `/login` `/register` 上验证 `as_token`”的显式 contract。
2. 当前 appservice 结构化配置真相在 `DATA-D1-005`，并只保存 `as_token_secret_ref` 这类 secret reference；client ingress 并没有一个已登记的、gateway-visible token validation source。
3. namespace ownership 规则已在 `34` 中存在，但 `/register` / `/login` 的 client path 仍没有对应的 auth + namespace 裁决接口，因此也无法诚实实现 `M_EXCLUSIVE` / sender-localpart / user-namespace ownership。
4. 当前 `GET /login` / `GET /register` 只宣告 `m.login.password` / `m.login.dummy`；这是符合当前已实现 truth 的，但 application-service legacy auth 仍处于“官方要求已知、本地 contract 未闭合”的状态。

因此，`CS-GAP-002` 现在的真实问题不是“是否顺手补一个 errcode”，而是：

* appservice token 真值应该由哪个 authority 提供给 `gateway-worker`；
* 该 authority 如何在不泄漏 raw token 的前提下支持 `/login` `/register` 的 request-time validation；
* `/register inhibit_login=true` 是否在 `MX-AS-001` 未启用时仍需要完整支持，还是必须先通过新的 decision 缩小当前 product boundary。

## Why It Matters

* 如果没有 gateway-visible appservice auth contract，就无法把 invalid `as_token`、valid `as_token` + unsupported legacy login、以及 namespace ownership 失败这些官方分支正确地区分开。
* 如果没有 namespace ownership 裁决路径，就无法诚实实现 appservice register/login 所要求的 `M_EXCLUSIVE` 等语义。
* 当前项目已经把 appservice 主体能力登记为 `MX-AS-001` `Required-Conditional`；若继续让 client `/login` `/register` 对 `m.login.application_service` 保持隐式状态，会让 `MX-CS-002` 与 `MX-AS-001` 的边界漂移。
* `TEST-CS-001` / `EVID-CS-001` 也需要知道这类路径现在是否属于当前 release gate，还是必须等 appservice auth contract 进入 spec 后再纳入验证。

## Options

1. Define a gateway-visible appservice auth projection
   为 `gateway-worker` 增加已登记的 appservice token validation / namespace lookup contract，然后补齐 `/login` `/register` 的 official semantics。

2. Freeze the current product boundary by decision
   通过新的 `DEC-*` 明确声明当前 profile 不支持 application-service client auth，并规定对 `m.login.application_service` 的 deterministic reject wire truth。

3. Keep the gap open until `MX-AS-001` is promoted from config-only to client-auth-capable
   先不伪造 route behavior，只把当前缺失的 auth contract / namespace contract 明确落入 Spec + OQ + backlog。

## Current Recommendation

当前建议采用 Option 3：

* 先不在代码里假装实现 `m.login.application_service`；
* 先把 `30/12/23/43/44` 中相关边界写清，明确当前 discovery 不宣告该 flow，且原因不是“忘了实现”，而是 `OQ-0008` 尚未定义 appservice client-auth contract；
* 待 appservice token validation source、namespace ownership read path 与 current release boundary 明确后，再决定是进入完整实现还是通过 `DEC-*` 维持 deterministic reject。

## Closure Criteria

本问题关闭前，必须至少满足以下条件之一：

* 已在 Spec 中定义 `gateway-worker` 可用的 appservice token validation + namespace ownership contract，并据此实现 `/login` `/register` 的 official `m.login.application_service` semantics；
* 或项目通过新的 `DEC-*` 明确声明当前 profile 不支持该 flow，并把 deterministic wire truth、测试与 evidence 一并冻结；
* 并且 `TODO.matrix-client-server-gap-audit.md`、相关 `TEST-*` / `EVID-*`、以及主线 `TODO.md` 都已同步到真实状态。
