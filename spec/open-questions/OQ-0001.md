## Question

历史上，framework 层曾把若干 Matrix `v1.17` client-server route families 显式降级为 `Deferred` / `Required-Conditional`，并补上 deterministic stub guard contracts；当时尚未决定这些 surface 是应继续拆成 dedicated full contracts，还是应先固定为明确产品边界，包括但不限于：

* `GET /_matrix/client/*/login/sso/redirect`
* `GET /_matrix/client/*/login/sso/redirect/{idpId}`
* `POST /_matrix/client/v1/login/get_token`
* `GET /_matrix/client/v1/auth_metadata`
* `POST /_matrix/client/*/register/email/requestToken`
* `POST /_matrix/client/*/register/msisdn/requestToken`
* `POST /_matrix/client/*/account/password/email/requestToken`
* `POST /_matrix/client/*/account/password/msisdn/requestToken`
* `POST /_matrix/client/*/user/{userId}/openid/request_token`
* `GET /_matrix/client/*/voip/turnServer`
* `GET /_matrix/client/*/events`
* `GET /_matrix/client/*/events/{eventId}`
* `GET /_matrix/client/*/initialSync`
* `GET /_matrix/client/*/rooms/{roomId}/initialSync`
* `GET /_matrix/client/*/account/3pid`
* `POST /_matrix/client/*/account/3pid`
* `POST /_matrix/client/*/account/3pid/add`
* `POST /_matrix/client/*/account/3pid/bind`
* `POST /_matrix/client/*/account/3pid/delete`
* `POST /_matrix/client/*/account/3pid/email/requestToken`
* `POST /_matrix/client/*/account/3pid/msisdn/requestToken`
* `POST /_matrix/client/*/account/3pid/unbind`
* `GET /_matrix/client/*/joined_rooms`
* `GET /_matrix/client/*/notifications`
* `GET /_matrix/client/*/pushers`
* `POST /_matrix/client/*/pushers/set`
* `GET/PUT/DELETE /_matrix/client/*/directory/room/{roomAlias}`
* `GET /_matrix/client/*/rooms/{roomId}/aliases`
* `GET/PUT /_matrix/client/*/directory/list/room/{roomId}`
* `GET /_matrix/client/*/room_summary/{roomIdOrAlias}`
* `POST /_matrix/client/*/rooms/{roomId}/upgrade`
* `POST /_matrix/client/*/rooms/{roomId}/report` 与相关 reporting surfaces
* `GET /_matrix/client/*/media/preview_url`

该问题现已由 [DEC-0001](/root/Matrix/spec/decisions/DEC-0001.md) 收敛：上述 surface 在当前产品边界内统一固定为 stub-only / deterministic disabled truth。开发团队不得把它们视为“等待推导实现”的灰区，也不得把 `IF-CS-031`、`IF-CS-034`、`IF-CS-052` 这类已完整落位的 query family 再误记为本问题的一部分。需要特别说明的是，其中 `GET /_matrix/client/*/joined_rooms`、`GET /_matrix/client/*/notifications` 与 `GET /_matrix/client/*/rooms/{roomId}/initialSync` 在 Matrix `v1.17` 中仍是 current surfaces；这里的问题不是“上游是否已移除”，而是“本项目是否把它们纳入当前产品边界”。

## Why It Matters

* 当前 [12-matrix-protocol-compliance-profile.md](/root/Matrix/spec/framework/12-matrix-protocol-compliance-profile.md) 与 [23-interface-contract-catalog.md](/root/Matrix/spec/framework/23-interface-contract-catalog.md) 已避免对这些 route family 继续做隐式 `Required-Core` 声称，并补上 stub guards，但尚未把它们下钻为 dedicated full contracts。
* 若不把上述 surface 显式分解，团队会被迫回到 Matrix 官网推导行为，违反“仓库文档为唯一真相”的目标。
* 其中 SSO/login-token、email/SMS verification `requestToken`、OpenID delegation、TURN credentials、legacy event stream / account-wide initial sync / room-preview compatibility surface、3PID/identity-service、`/notifications`、`/pushers*`、`room alias/directory`、`room upgrade`、`preview_url` 都会引入额外数据模型或安全边界，不能只靠把 route 名称塞进 `IF` 表格解决。

## Options

1. Expand existing IF families
   把适合复用现有语义的查询类 surface 并入现有 `IF-CS-031`、`IF-CS-034`、`IF-CS-052`，同时补足数据与测试落位。

2. Introduce dedicated IF/DATA/FLOW entries
   为 SSO/login-token、3PID/identity-service、notifications、pushers、alias/directory、room upgrade、reporting、preview 专门新增契约族和数据契约，换取更清晰的实现边界。

3. Keep deterministic stubs until product-boundary decision
   在产品边界仍未决定或完整模型尚未落地期间，保留 `12/23/43/44` 中的显式降级与 stub wire behavior，避免继续隐式声称支持。

## Current Recommendation

本问题的历史建议已由 [DEC-0001](/root/Matrix/spec/decisions/DEC-0001.md) 取代。当前唯一有效结论是：

* 已经完整落位的 query family 保持现状，不再作为 open question 范围；
* 其余 surface 在当前产品边界内固定为 stub-only；
* 未来若要启用 full feature，必须先新增 dedicated sub-spec 与新的 `DEC-ID`，再修改 `12/23/43/44`。

## Closure Criteria

本问题已关闭，闭环条件如下：

* `affected IDs` 已收敛为当前真实受影响集合，并移除了已完整落位的 `IF-CS-031`、`IF-CS-034`、`IF-CS-052`；
* 每个不计划近期实现的 surface 都已有 [DEC-0001](/root/Matrix/spec/decisions/DEC-0001.md) 固化当前产品边界；
* `12/23/43/44` 已以 `DEC-0001` 作为现行边界真相，而不再依赖“以后再补”的开放性表述。
