## Question

当前 framework 层已把若干 Matrix `v1.17` client-server route families 显式降级为 `Deferred` / `Required-Conditional`，但这些 surface 仍未在 `IF/DATA/FLOW` 中形成可实现契约，包括但不限于：

* `GET /_matrix/client/*/joined_rooms`
* `GET /_matrix/client/*/notifications`
* `GET /_matrix/client/*/pushers`
* `POST /_matrix/client/*/pushers/set`
* `GET/PUT/DELETE /_matrix/client/*/directory/room/{roomAlias}`
* `GET /_matrix/client/*/rooms/{roomId}/aliases`
* `GET/PUT /_matrix/client/*/directory/list/room/{roomId}`
* `GET /_matrix/client/*/room_summary/{roomIdOrAlias}`
* `POST /_matrix/client/*/rooms/{roomId}/upgrade`
* `POST /_matrix/client/*/rooms/{roomId}/report` 与相关 reporting surfaces
* `GET /_matrix/client/*/media/preview_url`

这些 surface 目前虽然已不再被隐式当作 `Required-Core` 覆盖，但仍没有完整契约与实现级数据落位，因此“开发团队直接按 spec 编码”的目标在这些区域仍未成立。

## Why It Matters

* 当前 [12-matrix-protocol-compliance-profile.md](/root/Matrix/spec/framework/12-matrix-protocol-compliance-profile.md) 已避免对这些 route family 继续做隐式 `Required-Core` 声称，但尚未把它们下钻为 dedicated contracts。
* 若不把上述 surface 显式分解，团队会被迫回到 Matrix 官网推导行为，违反“仓库文档为唯一真相”的目标。
* 其中 `/notifications`、`/pushers*`、`room alias/directory`、`room upgrade`、`preview_url` 都会引入额外数据模型或安全边界，不能只靠把 route 名称塞进 `IF` 表格解决。

## Options

1. Expand existing IF families
   把适合复用现有语义的查询类 surface 并入现有 `IF-CS-031` / `034` / `052`，同时补足数据与测试落位。

2. Introduce dedicated IF/DATA/FLOW entries
   为 notifications、pushers、alias/directory、room upgrade、reporting、preview 专门新增契约族和数据契约，换取更清晰的实现边界。

3. Mark selected surfaces as explicitly deferred or conditional
   若产品边界近期不会实现其中部分 surface，则必须在 `12/23/43/44` 中显式降级，避免继续隐式声称支持。

## Current Recommendation

优先采用 “1 + 2” 的混合策略：

* 查询复用型 surface 进入既有 IF family；
* 会引入新 authority、派生索引或外部安全边界的 surface 建立 dedicated contracts；
* 在 dedicated contracts 未落地前，不允许继续隐式把这些 surface 视为已被 framework 完整覆盖。
