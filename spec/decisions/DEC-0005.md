## Decision

当前项目正式接受以下 production automation model 作为未来面向用户的唯一默认路径：

* `prod-install` 负责首次或幂等 bootstrap prod topology；
* `release-candidate` 负责从 reviewed non-local attestation 生成 `ReleaseCandidateManifest`；
* `promote-prod` 只允许消费 reviewed candidate manifest 做 production promotion；
* `rollback-prod` 只允许消费先前 `ProdPromotionRecord` 中已经写出的 rollback handle；
* `prod-cost-monthly` 继续独立负责 `ProdCostSnapshotAttestation`，不得与日常 promote 混写。

本决策同时固定以下边界：

* production promote 不得直接把“当前 pushed head”或“某个 release ref”当作已验证候选；必须先有 `ReleaseCandidateManifest`。
* `requires_do_migration = false` 的 gateway promote 默认走 Cloudflare versions/deployments gradual path；`requires_do_migration = true` 或 prod 尚未 bootstrap 时，必须改走 `wrangler deploy` 路径，且不得伪装成可一键 rollback 的 gradual release。
* `prod-install` / `promote-prod` / `rollback-prod` 产物必须分别落成 `ProdInstallRecord` / `ProdPromotionRecord` / `ProdRollbackRecord`，并回链 GitHub run identity 与 Cloudflare deployment identity。
* 本决策不会关闭 [OQ-0002](/root/Matrix/spec/open-questions/OQ-0002.md)。即使 prod topology automation 落地，只要真实 closed-month production snapshot 尚未形成，`EVID-COST-001` 仍不得转为 `pass`。

## Rationale

* 用户日常真正需要的是“一次安装、一次批准、失败可回滚”，而不是每次发版都重新拼 Cloudflare 资源与 Wrangler 参数。
* 把 install / promote / rollback / monthly cost 拆开后，才能同时满足低摩擦、可审计和 fail-closed；若把它们混成一个 workflow，失败域、审批语义与成本窗口都会互相污染。
* 当前仓库已有可信的 non-local attestation contract，但还没有 production candidate contract；若不先钉死 reviewed candidate model，后续很容易退化为“拿最新 HEAD 直接上 prod”。
* 当前 remaining L1 blocker 仍然是 cost-only gates；本决策只收敛 production automation contract，不虚报 `OQ-0002` 或 `L1 Local-Core` 已闭环。

## Consequences

* `42`,`24`,`25`,`26`,`43`,`TODO.md` 必须同步落下 production automation contract；
* 后续 workflow / CLI / local tests 必须围绕 `ReleaseCandidateManifest`、`ProdInstallRecord`、`ProdPromotionRecord`、`ProdRollbackRecord` 实现；
* production topology automation 可以帮助收口 `OQ-0002` 中“target account 尚不存在规范固定命名 prod topology”这一半 blocker；
* 但 closed-month snapshot 与 pre-release official cost proof 仍分别受 `OQ-0002` / `OQ-0006` 约束，不得因本决策而降标。
