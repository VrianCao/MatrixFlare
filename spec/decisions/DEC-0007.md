## Decision

`GET /_matrix/client/versions` 在当前项目中必须返回一条 cumulative stable version ladder，至少包括 `r0.6.1` 与 `v1.1` 到 `v1.17`，以满足当前官方 Matrix browser client 的发现兼容基线。

但这条 ladder 只解决 browser compatibility discovery，不会自动放宽任何已被 `DEC-0001` 固化为 stub-only / deterministic disabled truth 的 surface：

* `/_matrix/client/versions` 的 ladder 不能被解释为“当前仓库已完整启用所有历史 stable 版本中的全部可选能力”；
* 具体 capability、stub-only、unsupported、discoverability truth 与 route behavior 仍然只由现行 `MX-*`、`IF-*`、`TEST-*`、`EVID-*` 与 `DEC-0001` 决定；
* 若某 browser client 因 `/versions` 接受本 homeserver，后续仍必须通过现行 capability truth、stub truth 与 deterministic error model 观察到真实产品边界，而不是被便利性伪装成 full feature。

对 production automation 的 readiness 口径还必须再区分“中间 rollout step”与“最终 browser-facing cutover”：

* `jobs-worker` / `ops-worker` 的单独切换，以及 `gateway-worker` gradual rollout 中 `0% < candidate < 100%` 的中间阶段，只能证明某一段 production rollout 仍在推进，不能被解释为“当前 browser-facing prod gateway 已整体满足官方浏览器发现基线”；
* 因此这类中间阶段的 worker-scoped readiness probe 可以暂时不把 `/versions` ladder 当作通过前提；
* 但一旦 probe 试图证明最终 browser-facing gateway 已 ready，例如 `gateway-worker` 已切到 `100%`、generic browser smoke 指向当前 active prod ingress、或任何 release/evidence gate 要消费该部署时，就必须继续对缺失 cumulative ladder 的 `/versions` fail-closed。

## Rationale

* 当前官方 Matrix browser client 基线来自 `matrix-js-sdk`，其版本兼容判断仍要求 homeserver 宣告与 `v1.1` 到 `v1.13` 的交集；只返回 `v1.17` 会在登录前被误判为不兼容。
* Matrix 规范的 `/versions` 对客户端 discovery 有规范性意义，因此仓库必须显式记录：这里解决的是客户端兼容发现问题，不是产品边界扩张。
* 若不冻结这个解释，`MX-CS-001` 的热修复会与 `DEC-0001` 的 stub-only 边界形成可审计歧义。

## Consequences

* `12`,`25`,`30`,`43`,`44`,`15`,`TODO.md` 必须把 `/versions` ladder 明确写成 browser compatibility discovery contract，而不是笼统的 feature enablement claim；
* `TEST-CS-001` / `EVID-CS-001` 必须证明 ladder 真实存在；
* readiness gate 不得再把“中间 rollout step 的暂时放宽”误写成最终 browser-facing prod ready；对最终 gateway cutover、generic remote smoke 与 evidence gate，仍不得把只返回 `["v1.17"]` 的部署视为通过；
* 若未来要改变 `/versions` ladder 的语义或范围，必须更新本决策及相关 `MX/TEST/EVID`，而不是只改实现。
