## Decision

以下 client route families 在当前产品边界内固定为 stub-only / deterministic disabled truth，而不是“等待开发团队自行补完 full feature”的半开放区域：

* SSO redirect / login-token / auth metadata
* 3PID / identity-service-assisted client flows
* email / msisdn `requestToken` bootstrap
* pushers / external push gateway integration
* URL preview
* reporting APIs
* joined rooms / room alias directory / room summary
* notifications listing endpoint
* room upgrade
* OpenID request-token delegation
* VoIP TURN credentials
* legacy initial sync / deprecated event stream routes

当前规范真相固定如下：

* `12-matrix-protocol-compliance-profile.md` 中这些 surface 的 `Support Level` 与 notes，只表达“当前 release profiles 下必须维持 deterministic stub truth”，不表达“允许开发团队自行脑补 full feature”。
* `23-interface-contract-catalog.md` 中对应 `IF-CS-*` 行是当前唯一可实施 contract；实现必须按固定错误体、固定短路优先级、固定 discoverability truth 落地。
* `43-testing-and-compliance.md` 与 `44-verification-and-evidence-register.md` 中的 `TEST-CS-004` / `EVID-CS-004` 是这些 surface 在当前产品边界内 stub-only 行为的专属发布门禁；跨域安全、治理与 profile 级附加门禁仍按各自 `MX/TEST/EVID` 生效。
* 若未来要把其中任一 surface 从 stub-only 改为 enabled feature，必须先新增 dedicated `IF/DATA/FLOW/TEST/EVID` 子规范，并同时新增或替换本决策；仅修改 feature flag 或局部 handler 不构成有效启用。

## Rationale

* 这些 surface 普遍跨外部认证、身份、通知、TURN、SSRF、兼容性或产品运营边界，不能留给实现阶段“顺手补”。
* 对开发团队而言，“明确拒绝且如何拒绝”本身就是可编码真相；把它们继续保留为开放问题，会削弱“文档即唯一真相”。
* 当前仓库中与 search / room query 相关的 `IF-CS-031`、`IF-CS-034`、`IF-CS-052` 已有 dedicated contracts，不应再被误归为同一开放问题。

## Consequences

* 当前 release profiles 下，实现这些 route family 的正确行为是 deterministic stub，而不是 full feature。
* 任何试图绕过 dedicated sub-spec 直接启用 handler 的改动，都必须被视为规范违例。
* `OQ-0001` 关闭，相关摘要寄存器改为引用本决策。
