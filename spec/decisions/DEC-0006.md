## Decision

当仓库已经存在真实 prod topology，但 `release-candidate -> promote-prod` 又因 `OQ-0002` / `OQ-0006` 的成本门禁无法生成同 head reviewed candidate，导致项目无法把新代码部署到 prod、也就无法积累关闭成本 blocker 所需的真实使用窗口时，仓库接受一条受控例外路径：

* 允许新增 `operational-prod-refresh` GitHub Actions workflow；
* 它只允许在已有有效 `ProdInstallRecord` / `ProdPromotionRecord` baseline 的情况下运行；
* 它只允许把当前 checked-out head 直接刷新到现有 prod topology，并保留 raw state、R2 object、deployment identity、readiness snapshots 与 rollback handle；
* 它必须把此次部署明确标记为 `promotion_authority = operational_unblock`，并写明这是为了解开 `OQ-0002` / `OQ-0006` 的 Phase 08 cost deadlock；
* 它不得生成 `ReleaseCandidateManifest`，也不得被当作 `TEST-OPS-003` / `EVID-OPS-003` / `L1 Local-Core` 已闭环的证明。

## Rationale

* 当前 Phase 08 剩余 blocker 已收敛到真实成本证据；若 prod 永远不允许先上线给真实用户使用，就永远拿不到关闭 `OQ-0002` 所需的 post-install closed billing period。
* 继续把正式 reviewed-candidate 流程与“为了收集真实成本数据而先运行 prod”强耦合，会形成结构性死锁。
* 该例外路径保留了 deployment identity、rollback handle、readiness probe 与 artifact provenance，因此它仍然是可审计的运营动作，而不是口头越权。

## Consequences

* `42`,`24`,`26`,`TODO.md` 必须同步记录 `operational-prod-refresh` 的边界；
* `ProdPromotionRecord` 必须能区分 `reviewed_candidate` 与 `operational_unblock` 两种 authority；
* 正式 `release-candidate` / `promote-prod` 仍然是未来面向用户的默认发布路径，不被本决策替代；
* 本决策不会关闭 `OQ-0002`、`OQ-0006`，也不会把任何 Phase 08 / L1 claim 提前转为完成。
